# 生产数据库方案评估

## 当前方案：SQL.js

**优点：**
- 零依赖，无需安装数据库服务
- 一键启动，适合原型和本地开发
- 数据持久化为单一文件，备份简单

**缺点（生产环境）：**
- 每次写操作触发全量导出，高并发下性能瓶颈
- 无真正的事务支持，扣费等关键操作无法原子化
- 内存数据库 + 文件持久化模型，进程崩溃可能丢数据
- 不支持多实例共享，无法水平扩展
- 并发写入存在竞争风险

## 推荐方案

### 首选：PostgreSQL + Prisma / Knex

| 方面 | 说明 |
|------|------|
| 成熟度 | 行业标准，社区庞大 |
| 事务 | 完整的 ACID 支持，关键计费场景必须 |
| 并发 | MVCC 模型，高并发读写无锁竞争 |
| 扩展 | 支持读写分离、连接池、多实例 |
| 本地开发 | `docker compose up postgres` 一条命令 |
| 迁移成本 | 中等 — 需改写 SQL 为 Knex/Prisma 查询，约 3-5 天 |

### 备选：better-sqlite3 (原生 SQLite 驱动)

| 方面 | 说明 |
|------|------|
| 优势 | 比 sql.js 快 10-50 倍，原生 C 驱动，支持 WAL 模式 |
| 事务 | SQLite 支持事务（需手动 BEGIN/COMMIT） |
| 迁移成本 | 低 — SQL 语法完全兼容，只需改 `require('sql.js')` → `require('better-sqlite3')` |
| 上限 | 仍是单机方案，不适合多实例部署 |

## 建议

**短期（上线前）：** 迁移到 `better-sqlite3`，成本最低，性能提升明显，可支持数百并发用户。

**长期（规模化后）：** 迁移到 PostgreSQL，适用于千级以上用户或需要多实例部署的场景。
